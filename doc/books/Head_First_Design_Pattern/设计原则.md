# 设计原则

## OO 的 4 个基本概念

## OO 的设计原则（Principle）

原则很重要，每个设计模式包含了几个 OO 原则。  
OO 原则是目标，设计模式是做法。  
尽量达到这些原则，而不是随时要遵守这些原则。

OO 的设计原则：

### 原则 1：封装变化。

> 找出应用中可能需要变化之处，把他们独立出来并封装，不要和那些不需要变化的代码混在一起，好让其他部分不会受到影响。

分开变化和不变化的部分->让 系统中某部分变化不影响其他部分：  
如果要增加新的需求，都会使得某方面的代码发生变化，那么可以确定：这部分代码需要被抽出来，和其他稳定的代码区分开来。
把会变化的部分取出并封装出来，是为了可以轻易地改动或扩充次部分，而不影响不需要变化的其他部分。

### 原则 2：针对接口编程，而不是针对实现编程。

由行为类而不是 Duck 类实现接口行为。
针对接口编程的关键在于多态：变量的声明类型应该是超类型，通常是一个抽象类或者是一个接口。
利用多态，针对超类型编程，执行时会根据实际情况执行真正的行为，不会绑定到超类型的行为。

### 原则 3：多用组合（composition），少用继承。

组合不同于继承的地方：鸭子的行为不是继承而来的，而是和适当的行为对象“组合”来的。  
为了实现“在运行时动态地改变行为”，提供相应的 set 方法。  
使用组合建立系统具有弹性，不仅仅将算法族封装成类，更可以“在运行时动态地改变行为”，只要组合的行为对象符合正确的接口标准即可。

- 类之间的关系  
  IS-A（是一个）：继承  
  HAS-A（有一个）：组合  
  IMPLEMENTS（实现）
- “有一个”可能比“是一个”更好

### 原则 5：依赖倒置原则（Dependency Inversion Principle）：依赖抽象，不要依赖具体类。

说明：【原则 5：依赖倒置原则】  
不管高层组件或低层组件，两者都应该依赖抽象类，不能依赖具体类。不能让高层组件依赖低层组件。  
最有威力的技巧：工厂模型。但不是唯一的技巧。  
（1）帮助遵循此原则的指导方针：  
变量不可以持有具体类的引用。要用工厂模式来避开。  
不要让类派生自具体类。派生一个抽象（接口或者抽象类）  
不要覆盖基类中已实现的方法。设计一个基类不要子类来覆盖。当然有时需要覆盖。

原则和模式可以应用在软件生命周期的任何阶段。  
封装行为时，常用的类之间“关系”：IS-A，Has-A，或 IMPLEMENTS（实现）。

软件开发完成“前”：致力于提高开发速度。  
软件开发完成“后”：致力于提高可维护性和可扩展性的复用程度。

使用设计模式共享词汇进行沟通的优点？  
① 共享模式词汇的威力强大，交流的不仅是模式名称而已，而是一整套模式背后所象征的质量、特性和约束。  
② 使得用更少的词汇做更充分的沟通。  
③ 使得思考架构的层次提高到模式层和设计层，而不是停留在细节。  
④ 共享模式词汇可帮助团队成员快速充电。  
⑤ 共享模式词汇可帮助初级开发人员迅速成长。

如何使用设计模式？  
我们全部使用别人设计好的库和框架。选择合适的库和框架，并放入合适的地方。但是，库和和框架无法帮助我们将应用组织成容易了解、容易维护、具有弹性的架构，所以需要设计模式。

设计模式用来组织类和对象以解决某些问题。  
库和框架不是设计模式。了解设计模式，会更容易了解这些 API 如何围绕着设计模式来构造的。  
设计时，弹性的设计，可以维护，可以应付变化。

OO 基础：抽象、封装、继承和多态。  
OO 原则：3 个。  
OO 模式：  
第 1 个模式：策略模式（Strategy Pattern）。

良好的 OO 设计必须是具备可复用、可扩充、可维护三个特征。  
模式可以让设计出具有良好 OO 设计质量的系统。  
模式被认为历经经验的 OO 设计经验。  
模式不是代码，而是针对设计问题的通用解决方案。  
模式不是被发明，而是被发现。  
大多数的模式和原则，都是着眼于软件变化的主题。  
大多数的模式都允许系统局部改变独立于其他部分。  
常常把系统中变化的部分抽出来封装。  
模式让开发人员之间有共享的语言，能够最大化沟通的价值。
